## はじめに
2020年頃から細々とPostgreSQLへのコントリビューションを行っていたのですが、今年はいつもよりも頑張るぞ！と勝手に意気込んでいたので、その成果として2025年に私が開発したPostgreSQLの機能について解説したいと思います！

## pg_dumpで取得したバックアップのリストア時にpsqlの設定をデフォルト推奨とするドキュメントを追加
- タイトル：Doc: recommend "psql -X" for restoring pg_dump scripts.
- コミッター：Tom Lane
- 日時：Sat, 25 Jan 2025
- リンク：https://git.postgresql.org/gitweb/?p=postgresql.git;a=commit;h=d83a108c10a3ec886a24c620a915aa2c5bc023aa

この変更はpg_dumpで取得したダンプファイルをpsqlで読み込むとき、psqlがデフォルト以外の設定（自分が確認したのはAUTOCOMMIT=off）だとエラーになるときがあるのに対処したものになります。対処といってもドキュメントの修正で、psqlの`-X` (`--no-psqlrc`)オプションを使うことで、psqlの設定ファイルを読み込まないようにすることを推奨するものとなります。

psqlの設定ファイル`psqlrc`については過去の自分のブログをご参照ください。
https://zenn.dev/shinyakato/articles/543dae5d2825ee

`\set AUTOCOMMIT off`を設定したうえで、以下のケースを検証します。
```bash
-- 適当なデータベースを作成する
$ createdb test1

-- pg_dumpallでSQLスクリプトファイルをダンプ
-- -cオプションは、ダンプされたデータベース、ロール、テーブル空間を再作成する前に、
-- それらをすべてDROPするSQL文を発行する
$ pg_dumpall -c -f test1.sql

-- psqlでSQLスクリプトファイルをリストア
$ psql -f test1.sql
~snip~
psql:test1.sql:14: ERROR:  DROP DATABASE cannot run inside a transaction block
psql:test1.sql:23: ERROR:  current transaction is aborted, commands ignored until end of transaction block
psql:test1.sql:30: ERROR:  current transaction is aborted, commands ignored until end of transaction block
psql:test1.sql:31: ERROR:  current transaction is aborted, commands ignored until end of transaction block
~snip~
```

`-c`により生成される`DROP DATABASE`文は、トランザクションブロック内で実行できないSQL文であるため、このようなエラーが発生します。また、デフォルトではトランザクションブロック内の文がエラーになると、トランザクション全体がアボートするため、後続のSQL文も失敗してしまいます。

他にもトランザクションブロック内で実行できないSQL文もあるので、それらをリストアするときには同様のエラーが発生します。

## pg_stat_progress_basebackupビューにbackup_typeカラムを追加
- タイトル：Add backup_type column to pg_stat_progress_basebackup.
- コミッター：Masahiko Sawada
- 日時：Tue, 5 Aug 2025
- リンク：https://git.postgresql.org/gitweb/?p=postgresql.git;a=commit;h=deb674454c5cb7ecabecee2e04ca929eee570df4

PostgreSQL 17からpg_basebackupに増分バックアップの機能が追加されました。しかし、バックアップの進捗確認を行うためのpg_stat_progress_basebackupビューには、フルバックアップなのか増分バックアップなのかを判別するためのカラムがありませんでした。

そこで、`backup_type`カラムを追加し、`full`（フルバックアップ）と`incremental`（増分バックアップ）を表示するようにしました。

pg_basebackupを実行し、別のターミナルからpg_stat_progress_basebackupビューを見てみます。

```bash
$ pg_basebackup -D full
```
`backup_type`に`full`が表示されていることがわかります。
```
=# TABLE pg_stat_progress_basebackup;
-[ RECORD 1 ]--------+-------------------------
pid                  | 853626
phase                | streaming database files
backup_total         | 1592460800
backup_streamed      | 622124544
tablespaces_total    | 1
tablespaces_streamed | 0
backup_type          | full                     <- new!
```

次に、pg_basebackup -iで増分バックアップを取得してみます。
```bash
$ pg_basebackup -i full/backup_manifest -D incl
```
`backup_type`に`incremental`が表示されていることがわかります。
```
=# TABLE pg_stat_progress_basebackup;
-[ RECORD 1 ]--------+-------------------------
pid                  | 854435
phase                | streaming database files
backup_total         | 1613615104
backup_streamed      | 726617088
tablespaces_total    | 1
tablespaces_streamed | 0
backup_type          | incremental              <- new!
```


## COPY FROMで複数行ヘッダーのファイルをサポート
- タイトル：Support multi-line headers in COPY FROM command.
- コミッター：Fujii Masao
- 日時：Thu, 3 Jul 2025
- リンク：https://git.postgresql.org/gitweb/?p=postgresql.git;a=commit;h=bc2f348e87c02de63647dbe290d64ff088880dbe

COPYコマンドには、`HEADER`というオプションがあります。これは、COPY FROMでデータをロードするときには、ヘッダー行をスキップするかどうかを制御でき、COPY TOでデータをエクスポートするときにテーブルのカラム名をヘッダとして出力するかを制御できます。しかし、COPY FROMで複数行ヘッダーのファイルをロードすることはできず、ヘッダー行をスキップするかどうかのbooleanしか指定できませんでした。

そのため、integerを受け付けるようにし、指定された行数分だけヘッダーをスキップできるようにしました。

```
=# \! cat /tmp/copy.csv
first header
second header
1,one
2,two
3,three

=# COPY t FROM '/tmp/copy.csv' WITH (HEADER 2, FORMAT csv);
COPY 3

=# TABLE t;
 id | data  
----+-------
  1 | one
  2 | two
  3 | three
(3 rows)
```

ちなみに、MySQL、SQL Server、Oracle SQL*Loaderでは同様の機能がすでにサポートされていました。
- MySQL: `LOAD DATA ... IGNORE N LINES` [^1]
- SQL Server: `BULK INSERT … WITH (FIRST ROW=N)` [^2]
- Oracle SQL*Loader: `sqlldr … SKIP=N` [^3]


## Autovacuumのログ出力を制御するパラメータをVACUUM用とANALYZE用に分離
- タイトル：Add log_autoanalyze_min_duration
- コミッター：Peter Eisentraut
- 日時：Wed, 15 Oct 2025
- リンク：https://git.postgresql.org/gitweb/?p=postgresql.git;a=commit;h=dd3ae378301f7e84c18f7a90f183c3cd4165c0da

これまでにも`log_autovacuum_min_duration`というパラメータがあり、これは、AutovacuumによるVACUUMとANALYZEの両方に対して、設定した以上の時間（デフォルト単位：ミリ秒）がかかった場合に、VACUUMまたはANALYZEのログを出力するというものでした。しかし、ANALYZEはテーブルのサンプリングの読み込みが中心である一方で、VACUUMはテーブルやインデックスのページの読み込みと書き込みの両方を行うので、VACUUMのほうが時間がかかる場合が多いです（もちろんテーブル定義、データサイズ、統計情報のサンプリング数、ワークロード、拡張統計の有無など様々な要因があるので一概には言えません）が、別々の設定を行うことができませんでした。

そこで、`log_autovacuum_min_duration`をVACUUM専用のパラメータとし、新たに追加した`log_autoanalyze_min_duration`をANALYZE専用のパラメータするようにしました。当初の提案では、他のパラメータ名との一貫性で、`log_autovacuum_vacuum_min_duration`と`log_autovacuum_analyze_min_duration`としていたのですが、既存のパラメータ名を変更するのは、後方互換性やpg_dump/pg_upgradeへの影響が大きく断念しました。

```
=# CREATE TABLE t (i int, d text) WITH (
  -- autoanalyze settings
  autovacuum_analyze_threshold = 1,
  autovacuum_analyze_scale_factor = 0,
  log_autoanalyze_min_duration = 0,
  -- autovacuum settings
  autovacuum_vacuum_threshold = 1,
  autovacuum_vacuum_scale_factor = 0,
  log_autovacuum_min_duration = 100_000_000
);
=# INSERT INTO t VALUES (1, 'a');
=# DELETE FROM t WHERE i = 1;
```
```

avg read rate: 18.229 MB/s, avg write rate: 0.000 MB/s
buffer usage: 155 hits, 7 reads, 0 dirtied
WAL usage: 1 records, 0 full page images, 530 bytes, 0 buffers full
```

## WALの圧縮後フルページイメージのサイズをpg_stat_walビューに表示
- タイトル：Add wal_fpi_bytes to pg_stat_wal and pg_stat_get_backend_wal()
- コミッター：Michael Paquier
- 日時：Tue, 28 Oct 2025
- リンク：https://git.postgresql.org/gitweb/?p=postgresql.git;a=commit;h=f9a09aa2952039a9956b44d929b9df74d62a4cd4

WAL圧縮（`wal_compression`）を有効化していると、WALのフルページイメージ[^4]が圧縮されるのですが、これまでは圧縮の効果を測るには以下の方法を取る必要がありました。

1. WAL圧縮有効化/無効化した設定で同じベンチマークを実行し、pg_stat_walビューの`wal_bytes`を比べることでどの程度圧縮が効いているかがわかります。ただし、この値は、生成された全WALの総量の比なので正確には圧縮率ではありません。（フルページイメージ以外のWALも含まれた状態での計算になる）
2. pg_waldump --fullpageをサーバ上で実行すると、圧縮後のWALのサイズがわかるので圧縮率を計算することができます。しかし、これを取得するにはWALファイルが必要であり、サーバ上での実行が必要なため、クラウドなどのサーバ上にログインできないような環境では使用できませんでした。（pg_wal_inspectが使用できればクラウドでも計算できるかもしれません。未確認。）

そこで、pg_stat_walビューに圧縮後のフルページイメージのサイズ`wal_fpi_bytes`を追加しました。

```
=# TABLE pg_stat_wal;
-[ RECORD 1 ]----+-----------------------------
wal_records      | 2031667
wal_fpi          | 288581
wal_bytes        | 6346674376
wal_fpi_bytes    | 1932610356                   <- new!
wal_buffers_full | 424447
stats_reset      | 2025-12-02 19:31:44.16184+09
```

このパッチで追加した情報を、EXPLAIN (WAL)やVACUUMのログに追加するパッチもコミットされています。[^5][^6]

## pg_stat_progress_vacuumビューにmodeカラムとstarted_byカラムを追加
- タイトル：Add mode and started_by columns to pg_stat_progress_vacuum view.
- コミッター：Masahiko Sawada
- 日時：Tue, 9 Dec 2025
- リンク：https://git.postgresql.org/gitweb/?p=postgresql.git;a=commit;h=f9a09aa2952039a9956b44d929b9df74d62a4cd4

VACUUMには、様々な起動理由（自動、手動、XID周回防止）があったり、様々な動作モード（通常、積極的、フェイルセーフ）がありますが、これらの情報を、VACUUMの進捗確認を行うためのpg_stat_progress_vacuumビューでは確認することができませんでした。

そこで、`mode`カラム（`normarl`、`aggressive`、`failsafe`）と`started_by`カラム（`manual`、`autovacuum`、`autovacuum_wraparound`）を追加しました。

pg_stat_progress_vacuumビューを見てみます。

```
=# TABLE pg_stat_progress_vacuum;
-[ RECORD 1 ]--------+--------------
pid                  | 362895
datid                | 5
datname              | postgres
relid                | 24602
phase                | scanning heap
heap_blks_total      | 8850
heap_blks_scanned    | 5327
heap_blks_vacuumed   | 0
index_vacuum_count   | 0
max_dead_tuple_bytes | 67108864
dead_tuple_bytes     | 0
num_dead_item_ids    | 0
indexes_total        | 0
indexes_processed    | 0
delay_time           | 0
mode                 | normal       <- new!
started_by           | autovacuum   <- new!
```

同様に、pg_stat_progress_analyzeビューに`started_by`カラム（`manual`、`autovacuum`）を追加するパッチもコミットされています。[^7]
